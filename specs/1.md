# Squid Coder – Product & Dev Spec (v3)

## 0. Goals & Constraints

**Product Goal:** Evolve Squid Coder from a simple AI/chat-based code generator into a developer-focused app builder with clear value, while fixing core billing and UX issues.

**Engineering Constraints:**

- Build on the existing stack and architecture.
- Ship in small, testable increments.
- Validate new concepts (planning/architecture) before investing in complex UIs.

Roadmap phases:

1. Fix – Core bugs and revenue blockers (P0)  
2. Polish – Dashboard UX and onboarding (P1)  
3. Validate – Lightweight planning step (P2)  
4. Scale – Optional visual architecture editor (P3, conditional)

---

## Phase 1: Fix (P0) – Core Bugs & Billing

### 1.1 Subscription & Billing Behavior (P0)

**Problem:** Subscriptions do not correctly persist plan/credits state. Only credit pricing and deduction appear consistent; plan and feature state are unreliable.

**Requirements:**

- Establish a single source of truth for:
  - User plan (Free, Pro, Pro Plus)
  - Monthly credit allowance
  - Plan-based feature flags (e.g., premium models)
- On subscription creation/change:
  - Persist active plan tier/ID.
  - Initialize or update monthly credit allowance.
  - Enable/disable premium models and features based on plan.
- On subscription expiration/downgrade:
  - Update plan state.
  - Adjust features/model access accordingly.

**Implementation Notes:**

- Use subscription table/model as canonical for plan.
- Fix logic for:
  - `currentPlan`
  - `creditsRemaining`
  - `hasPremiumModelAccess`
- Ensure billing events (webhooks/callbacks) reliably sync to app state.

---

### 1.2 Chat Error Handling (P0)

**Problem:** After an error, chats sometimes stop responding to further messages.

**Requirements:**

- On chat request failure:
  - Show clear inline error message in the chat UI.
  - Provide a “Retry” action for the failed message.
  - Allow sending new messages without page reload.
- Reset error/loading state on success or failure.

**Implementation Notes:**

- Audit chat flow (frontend + API) for:
  - Unhandled rejections
  - Stale loading/error flags
- Ensure cleanup via a `finally`-style path.

---

### 1.3 UI Polish – Separate Tickets (P1 but small / non-blocking)

Create **three independent tickets** for these; they are not blocking but easy wins.

#### 1.3.a Chat Input Focus Style

**Problem:** Focus outline appears on the textarea instead of the parent chat component.

**Requirements:**

- Remove default focus outline from the `<textarea>`.
- Apply focus styling (border, shadow, etc.) to the parent chat input container, matching URL input focus style.
- Preserve visible focus for accessibility.

---

#### 1.3.b URL Input Icon

**Problem:** URL input uses a rotating sparkle icon that’s distracting.

**Requirements:**

- Remove rotating sparkle animation from URL input.
- Either:
  - No icon, or
  - A static icon aligned with the design system.

---

#### 1.3.c Delete Modal Text Layout

**Problem:** Delete modal text has poor wrapping or alignment.

**Requirements:**

- Ensure text wraps correctly within modal width, without overflow.
- Align text per existing modal conventions (e.g., left-aligned body text).
- Verify behavior at multiple viewport sizes.

## Phase 2: Polish (P1) – Dashboard UX & Onboarding

### 2.1 Dashboard Information Architecture

**Goal:** The dashboard should answer:

- “What can I build?”
- “What have I already built?”
- “What should I do next?”

**Requirements:**

- Visually distinct sections:

1. **Hero / Intro**
   - Short headline + description emphasizing “AI app builder for developers.”
   - Primary CTA: “Create a new app.”

2. **Projects (Primary)**
   - Central section of the dashboard.
   - For each project card:
     - Name
     - Short description/brief
     - Current generation status (see Phase 3 for plan/code flags)
     - Status indicator (e.g., In progress, Complete)
     - Last updated timestamp including time, not just date.

3. **Milestones & Credits**
   - Compact summary of progress and usage.

4. **Plans / Upgrade (Secondary)**
   - Move full pricing details to a secondary “Upgrade” page or visually de-emphasize.
   - Still show:
     - Current plan
     - Upgrade CTA

---

### 2.2 Milestones & Credits UX

**Note:** Only invest heavily here if data shows users engage with milestones (see UnlockProgress usage). If engagement is low, deprioritize detailed milestone work.

**Requirements (if prioritized):**

- Milestones:
  - Show explicit steps (e.g., “Create first app”, “Generate first code”, “Deploy/export first project”).
  - Display:
    - Current progress (e.g., “1 of 4”)
    - Next milestone and its reward (e.g., bonus credits).

- Credits:
  - Next to credits count:
    - Tooltip or inline text explaining:
      - What credits are used for (generations).
      - Plan allowance and current usage (e.g., “Free plan includes X, you’ve used Y.”).

---

### 2.3 Onboarding & Help

**Goal:** Improve activation and feature discovery.

**Requirements:**

- First-time guided onboarding modal:
  - Trigger on first visit or first project creation.
  - Simple **3-step** flow:
    1. Create a project and write a brief.
    2. System plans/understands your app (planning step).
    3. System generates code; you iterate from there.

- “Help / Examples” panel:
  - Collapsible panel from dashboard or project view.
  - Contains:
    - Example prompts/briefs that perform well.
    - Short descriptions or screenshots of generated apps.
    - Quick tips for writing good briefs.

---

### 2.4 Accessibility

**Requirements:**

- Strong color contrast in both light and dark themes.
- Full keyboard operability for primary actions and navigation.
- ARIA roles/labels for:
  - Navigation
  - Sections
  - Interactive components
- Descriptive labels on key buttons/links:
  - Prefer “Continue building: [ProjectName]” over generic repeated text.

## Phase 3: Validate (P2) – Planning Step & Simple State

### 3.1 “Review Plan Before Generating” Flow

**Goal:** Validate whether a planning step improves outcomes, using existing prompts and minimal schema changes.

**Requirements:**

- Reuse the existing “software architect” prompt to generate a structured textual plan:
  - Routes/pages
  - Components
  - APIs
  - Data models
  - Key behaviors

- Flow for “High quality” (or equivalent) generations:
  1. User submits or edits project brief.
  2. System generates a **plan**.
  3. Show the plan in a structured, readable format (sections, bullets).
  4. Provide actions:
     - “Approve & generate code”
     - “Edit brief and regenerate plan”

- Persistence:
  - Add to the relevant model (e.g., Chat or Project):
    - `plan: string | null`
    - `hasPlan: boolean` (derived from `plan != null` is fine)
    - `hasCode: boolean` (set when code generation completes)
  - Option: store only `plan` and derive `hasPlan`. `hasCode` can be separate.

**Implementation Note:**  
Use **option 1** for storing the plan: add a `plan` field to the existing model (e.g., Chat), populate during planning, and clear/regenerate on brief edits.

---

### 3.2 Simple Status Flags Instead of Full Stage Enum

**Goal:** Track whether a project has a plan and code, without over-engineering a stage system during validation.

**Requirements:**

- At minimum, track:
  - `hasPlan: boolean`
  - `hasCode: boolean`
- Use these flags to:
  - Show status in project cards (e.g., “Planned”, “Code generated”).
  - Drive simple UI states (e.g., “Generate plan”, “Generate code”, “Regenerate plan”).

**Non-goals in Phase 3:**

- No multi-agent orchestration system.
- No stage enums or complex pipelines.
- No visual architecture graph.

---

### 3.3 Metrics & Success Criteria

**Goal:** Decide objectively whether to invest in a visual architecture editor (Phase 4).

**Events to Track:**

- `plan_generated`
- `plan_viewed` (user actually opens/views the plan screen)
- `plan_approved`
- `plan_edited` (user edits brief and regenerates)
- `code_generated` (tag whether a plan was used: with_plan / without_plan)
- Optionally: `code_error` or retry rate after code generation.

**Success Criteria (example threshold):**

- Plan approval rate (plan_approved / plan_generated) > ~60%.
- Users who go through the planning step show:
  - Fewer code generation errors/retries, or
  - Higher completion/retention on projects vs. those who skip.

If these criteria are met, proceed to Phase 4 exploration.

---

## Phase 4: Scale (P3) – Optional Visual Architecture Editor

**Condition:** Only begin this phase if metrics from Phase 3 demonstrate strong plan usage and positive impact.

### 4.1 Visual Architecture View (Conceptual)

**High-level Concept:**

- Convert validated textual plans into a structured architecture model and visualize/edit it before code generation.

**Conceptual Requirements (to be specced separately if greenlit):**

- Represent architecture as graph/tree:
  - Nodes:
    - Pages/routes
    - Components
    - Data models
    - APIs
  - Edges:
    - Usage/relationship links between these entities.
- Visual editor:
  - Basic operations:
    - Rename nodes
    - Add/remove nodes
    - Edit simple properties (e.g., auth requirements)
- Code generation:
  - Consume the structured architecture model as the primary input when generating code.

This is a multi-week effort and depends on Phase 3 validation.

  - Requires a separate, detailed spec if Phase 3 metrics justify it.
